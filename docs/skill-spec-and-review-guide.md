# Skill 规格与审核指南

> 精简版，用于指定（编写）和审核 Claude Agent Skills。  
> 基于 Anthropic 官方 *The Complete Guide to Building Skills for Claude*。

---

## 1. 文件结构规范

```
skill-name/              ← kebab-case，无空格/下划线/大写
├── SKILL.md             ← 必须，大小写严格匹配
├── scripts/             ← 可执行脚本（Python/Bash）
├── references/          ← 按需加载的参考文档
├── assets/              ← 模板、配置、静态资源
└── ❌ README.md          ← 禁止放在 skill 目录内
```

---

## 2. YAML Frontmatter 规格

### 必填字段

| 字段 | 规则 |
|------|------|
| `name` | kebab-case，与目录名一致，禁止含 "claude"/"anthropic" |
| `description` | **三段式**：做什么 + 何时触发 + 关键能力。≤1024 字符，禁止 XML 标签 `< >` |

### 可选字段

| 字段 | 用途 |
|------|------|
| `allowed-tools` | MCP 工具白名单，格式 `mcp__server__tool` |
| `argument-hint` | 调用时的参数提示 |
| `compatibility` | 环境要求说明，≤500 字符 |
| `license` | 开源协议（MIT / Apache-2.0） |
| `metadata` | 自定义 KV（author / version / mcp-server） |

### Description 撰写公式

```
[做什么] + [触发短语列表] + [不适用场景（可选）]
```

**合格示例：**

```yaml
description: "从剧本批量生成角色、场景、道具资产。当用户要求批量生成资产、一键出图、
生成所有角色场景道具、重新生成指定资产时使用。"
```

**不合格示例：**

```yaml
# ❌ 过于笼统，缺少触发条件
description: "处理项目资源。"

# ❌ 只描述功能，不说明何时使用
description: "实现多层级资产管理系统。"
```

## 3. SKILL.md 正文结构

遵循 **渐进式加载** 三层设计：

| 层级 | 内容 | 加载时机 |
|------|------|---------|
| L1：Frontmatter | name + description | 始终在 system prompt |
| L2：SKILL.md 正文 | 完整指令与流程 | 匹配触发后加载 |
| L3：references/ | 详细文档与 API 参考 | 按需读取 |

### 推荐正文骨架

```markdown
# Skill 名称

一句话摘要。

## 前置检查
- 环境依赖
- 认证状态

## 主流程
### Step 1: ...
### Step 2: ...
### Step N: ...

## 错误处理
### 错误：[常见错误]
原因 → 解决方案

## 示例
用户说："..." → 执行动作 → 预期结果
```

### 正文编写原则

| 原则 | 做 | 不做 |
|------|----|------|
| 具体可执行 | `python3 scripts/validate.py --input {file}` | "验证一下数据" |
| 结构化 | 编号步骤 + 验证门控 | 大段散文 |
| 渐进式 | 核心指令留正文，详细参考移至 `references/` | 万字 SKILL.md |
| 关键指令突出 | `## CRITICAL` / `## 必须` 置顶 | 重要内容埋在中间 |
| 确定性校验 | 用脚本做校验，代码比语言指令可靠 | "请仔细检查" |

**正文体量指标**：SKILL.md 正文建议 ≤5000 词，超出部分拆入 `references/`。

## 4. 设计模式速查

| 模式 | 适用场景 | 核心技法 |
|------|---------|---------|
| **顺序编排** | 多步骤必须按序执行 | 显式步骤依赖 + 每步验证 + 失败回滚 |
| **多服务协调** | 跨多个 MCP/API | 分阶段 + 阶段间数据传递 + 集中错误处理 |
| **迭代精炼** | 输出质量需要迭代 | 初稿 → 校验脚本 → 修正循环 → 终止条件 |
| **上下文路由** | 同一目标，不同工具 | 决策树 + 回退方案 + 向用户解释选择 |
| **领域智能** | 需嵌入专业知识 | 规则前置 → 执行 → 审计追踪 |

## 5. 审核清单

### A. 结构合规（必须全部通过）

- [ ] 目录名 kebab-case
- [ ] `SKILL.md` 存在且大小写正确
- [ ] 无 `README.md` 在 skill 目录内
- [ ] Frontmatter 有 `---` 分隔符
- [ ] `name` 字段 kebab-case，与目录名一致
- [ ] `description` 包含 **做什么** + **何时触发**
- [ ] 无 XML 标签 `< >`
- [ ] `name` 不含 "claude" / "anthropic"

### B. 指令质量

- [ ] 步骤编号清晰，无歧义
- [ ] 每个脚本/工具调用给出具体命令和预期输出
- [ ] 关键校验用脚本实现而非自然语言描述
- [ ] 错误处理覆盖常见失败场景
- [ ] 包含至少一个端到端示例
- [ ] SKILL.md 正文 ≤5000 词，详细内容在 `references/`
- [ ] 路径引用使用变量（`$SKILL_DIR` / `${SKILL_DIR}`）

### C. 触发准确性

- [ ] **正向触发**：3+ 种自然表述能触发
- [ ] **同义触发**：换一种说法仍能触发
- [ ] **负向隔离**：无关话题不触发
- [ ] description 包含用户可能使用的关键词/短语
- [ ] 有范围限定（适用于 X，不适用于 Y）

### D. 健壮性

- [ ] 前置依赖检查（环境/认证/数据）
- [ ] 断点续传或幂等设计（长流程）
- [ ] MCP/API 调用有重试或降级策略
- [ ] 敏感操作前有确认机制

### E. 可组合性

- [ ] 不假设自己是唯一 skill
- [ ] 与其他 skill 无命名/路径冲突
- [ ] 跨 skill 数据传递使用约定格式（JSON/文件）

## 6. 常见问题诊断

| 症状 | 原因 | 修复 |
|------|------|------|
| 上传失败 "Could not find SKILL.md" | 文件名大小写错误 | 重命名为 `SKILL.md` |
| 上传失败 "Invalid frontmatter" | YAML 缺 `---` 分隔符或引号未闭合 | 检查 YAML 语法 |
| 不触发 | description 过于笼统 | 添加具体触发短语和关键词 |
| 过度触发 | description 过于宽泛 | 添加负向声明 + 限定作用域 |
| 加载但不遵循指令 | 指令过长/关键点被淹没 | 关键指令置顶 + 拆分至 references |
| 响应变慢 | 上下文过大 | 压缩正文 + 减少同时启用的 skill 数 |
| MCP 调用失败 | 工具名拼写/认证/连接 | 先脱离 skill 单独测试 MCP 工具 |

## 7. Description 模板

可直接复制使用的 description 编写模板：

```yaml
description: "[一句话说明功能]。当用户[触发场景1]、[触发场景2]、[触发场景3]时使用。
不适用于[排除场景]。"
```

**生产管线类（本项目适用）：**

```yaml
description: "[从X输入生成Y输出]，含[关键特性1]、[关键特性2]、[关键特性3]。
当用户要求[动作1]、[动作2]、[动作3]、[动作4]时使用。"
```
