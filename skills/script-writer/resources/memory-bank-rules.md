# 记忆库行为规则

本文件定义 AI 何时记录、记录什么、如何推断满意度。记忆库贯穿 S1-S9 全流程，自动记录用户的修改请求和反馈行为。

**存储位置**：记忆库文件统一保存在 Skill 的 `memory-banks/` 目录下，命名格式为 `{项目名}_{测试人}.md`，便于 Skill 负责人集中查看和跨测试员分析。

---

## 1. 触发条件（何时记录）

以下情况**必须**创建记忆库条目：

1. **内容修改**：用户要求修改已生成的交付物内容（如"把开场改成..."、"这段对白不要..."）
2. **不满反馈**：用户对生成结果表达不满或提出改进建议（如"这个不够好"、"感觉太平了"）
3. **案例参考**：用户提供参考案例要求按案例调整（如"参考XXX的风格"、"像XXX那样写"）
4. **重写要求**：用户要求重写/重做某部分（如"重写第3集"、"这段重来"）
5. **规则反馈**：用户对 Skill 规则本身提出修改建议（如"以后开场都应该用冲突直给"）

## 2. 不触发条件（不记录）

以下情况**不创建**记忆库条目：

1. **路由指令**：纯流程控制（"状态"、"下一步"、"跳转S3"、"继续"）
2. **初始创意输入**：S1 首次输入是起始素材，不是修改行为
3. **纯信息查询**：用户询问进度或状态（"现在到第几集了？"、"当前阶段是什么？"）
4. **确认操作**：用户对当前结果表示满意并推进（"好的"、"可以"、"下一步"）

---

## 3. 记录条目字段

每条记忆库记录包含以下字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| 编号 | M-001 递增格式 | M-001、M-002 |
| 时间 | YYYY-MM-DD HH:MM | 2025-01-15 14:30 |
| 阶段 | 当前所在的 S1-S9 阶段 | S7-逐层写作 |
| 类型 | 9 类修改类型代号（见下方） | REWRITE |
| 用户原话 | 用户输入的关键内容（原文摘录或精简引用） | "开场不要用系统空间" |
| 修改目标 | 被修改的交付物/章节/具体内容 | 第1集·开场结构（0-30秒） |
| 修改方案 | AI 采取的修改策略（1-2 句概括） | 将系统空间独白改为冲突直给 |
| 修改结果 | 具体做了什么改动（1-2 句概括） | 重写第1集前30秒 |
| 满意度 | 满意度状态标记（见推断规则） | ✅ 满意 |
| 备注 | 补充信息 | 用户提供了手写改写范例 |

---

## 4. 修改类型分类（9 类）

| 类型代号 | 类型名称 | 说明 | 典型触发语 |
|---------|---------|------|-----------|
| CONTENT | 内容修改 | 修改已生成内容的具体文字/情节 | "把这段改成..."、"这里应该是..." |
| STRUCT | 结构调整 | 修改故事结构、集数分配、节奏安排 | "前3集节奏太慢"、"高潮应该提前" |
| STYLE | 风格调整 | 修改语言风格、叙事风格、情感基调 | "语气太正式了"、"要更口语化" |
| CHAR | 角色调整 | 修改角色设定、对白风格、行为逻辑 | "女主不应该这么软弱"、"反派动机不够" |
| WORLD | 设定修改 | 修改世界观、金手指、规则设定 | "金手指太强了"、"这个设定不合理" |
| ADD | 新增内容 | 添加新角色、新情节、新伏笔 | "加一个闺蜜角色"、"这里加个伏笔" |
| DEL | 删除内容 | 移除不需要的内容 | "去掉这段"、"这个角色不要了" |
| REWRITE | 重写 | 对已有内容进行大幅重写 | "这段重写"、"全部重来" |
| RULE | 规则反馈 | 对 Skill 规则本身的修改建议 | "以后都应该..."、"规则里应该加上..." |

> **注意**：`RULE` 类型具有最高分析价值，Skill 负责人应重点关注此类记录。

---

## 5. 满意度推断规则

满意度由 AI 根据用户后续行为自动推断：

| 判断条件 | 满意度标记 | 说明 |
|---------|-----------|------|
| 用户明确说"好/可以/不错/OK/满意/就这样" | ✅ 满意 | 直接正面反馈 |
| 用户未评价但继续推进下一步 | ✅ 满意（推断） | 沉默即认可 |
| 用户在同一主题上追加修改 | ⚠️ 部分满意 | 需同时更新前一条记录的满意度 |
| 用户明确说"不行/不对/重做/还是不好/不是这个意思" | ❌ 不满意 | 直接负面反馈 |
| 修改刚完成，用户尚未反馈 | ⏳ 待确认 | 等待后续输入判断 |

**追溯更新规则**：

- 当推断出满意度时，**必须回溯更新**之前标记为 ⏳ 的相关条目
- 当用户追加修改时，**必须更新**前一条同主题记录为 ⚠️ 部分满意

---

## 6. 统计摘要维护规则

在记忆库文件（`memory-banks/{项目名}_{测试人}.md`）末尾的统计摘要区，每次新增或更新记录后**自动重新计算**：

### 统计维度

1. **按类型统计**：统计 9 种修改类型各自的出现次数和占比
2. **按阶段统计**：统计 S1-S9 各阶段的修改次数和占比
3. **满意度统计**：统计 4 种满意度状态的次数和占比
4. **高频修改模式**：提取出现 >= 2 次的同类修改，归纳为模式描述

### 高频模式提取规则

- 同一类型 + 同一阶段的修改出现 >= 2 次 → 提取为高频模式
- 格式：`[类型] [阶段] [模式描述] （N次）`
- 示例：`REWRITE S7 开场结构反复重写 （3次）`

---

## 7. 记录行为规范

1. **即时记录**：检测到触发条件后，在执行修改的同时创建记录
2. **原话保留**：用户原话字段尽量保留原文，过长时精简引用（加引号标注）
3. **编号连续**：编号全局递增，不因阶段切换重置
4. **不打扰用户**：记录过程完全静默，不向用户提示"已记录到记忆库"
5. **统计同步**：每次新增/更新记录后，同步更新统计摘要区
